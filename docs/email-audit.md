# Email audit log — server-side record of what actually went out

> _Last updated: 2026-05-13. Owner: social-svc._

## Why this exists

Before this change, the only record of "what emails did we send" lived in the
operator's browser localStorage at key `builderquest:admin:v1`. The Admin
Console's "Queue (latest 50)" rendered that array directly. Consequences:

- Two browsers, two queues. Clear site data, no audit trail.
- Anyone with `SMTP_USER` + `SMTP_PASSWORD` could bypass `social-svc` entirely
  and send as `learnai@useyl.com` with no trace on our side.
- We had no answer to "did the recipient actually get it?" — only to "did
  we tell our SPA to queue it?"

The server-side log closes the first gap. Stalwart's admin API (the upstream
SMTP server runs Stalwart Mail at `mail.cloud-claude.com`) closes the second.

## What's in scope (this PR)

| Concern | Mechanism | File |
|---|---|---|
| Persist every send the SPA submits | Append-only JSONL at `/data/email-log.jsonl` | `services/social-svc/src/email-log.ts` |
| Capture Stalwart's queue id at submit | Parse `info.response` for `Message queued as <id>` / `OK id=<id>` | `services/social-svc/src/email-log.ts:parseStalwartQueueId` |
| Request bounces and delays | RFC 3461 DSN: `MAIL FROM ENVID=<log-id> RET=HDRS` + `RCPT TO NOTIFY=FAILURE,DELAY`; bounce return-path = `SMTP_FROM_EMAIL` | `services/social-svc/src/email.ts` |
| Show submitted history in the admin tab | New `GET /v1/email/log` | `services/social-svc/src/app.ts` |
| Show real delivery state per message | New `GET /v1/email/log/:id/status` — proxies Stalwart `/api/queue/messages/{queueId}` behind a feature flag | `services/social-svc/src/{app,stalwart}.ts` |
| Stop the browser queue from looking authoritative | Relabel UI: existing localStorage table becomes "Drafts (this browser only)" | `app/src/admin/AdminEmails.tsx` |

## What's deliberately out of scope (with rationale)

- **Open-pixel tracking and click-rewriting.** Adding these to transactional,
  manually-flushed admin mail to a small known recipient list is a GDPR
  obligation (lawful basis, opt-out, privacy-policy disclosure) that's
  disproportionate to the value. Revisit when there's bulk lifecycle sending.
- **MDN read receipts (`Disposition-Notification-To`).** Real-world delivery
  rate is 2–10% (Gmail web suppresses, Apple Mail ignores, Outlook
  user-prompts). Worse than no signal because operators would treat absence
  as proof of non-read.
- **Inbound DSN ingestion (bounce mailbox poller).** DSN *headers* go on
  outgoing mail in this PR, so bounces will arrive in `learnai@useyl.com`.
  Parsing them into the log row (IMAP loop + MIME `message/delivery-status`
  + ENVID join) is a clean follow-up issue.
- **Forward detection.** There is no protocol-level signal. Repeated opens
  from different fingerprints would be a heuristic, but we already aren't
  doing open tracking.

## ID model

Three IDs flow through the system. Knowing which is which matters for any
future debugging.

```
┌────────────────────┐
│ envId (local row)  │  el-<base36-ts>-<hex4>   set by social-svc;
│                    │                          becomes RFC 3461 ENVID
│                    │                          on MAIL FROM
└──────┬─────────────┘
       │ POST /v1/email/send → nodemailer.sendMail(...)
       ▼
┌────────────────────┐
│ rfcMessageId       │  <UUID@cloud-claude.com> generated by nodemailer
│                    │                          (or set explicitly by us);
│                    │                          travels in the Message-ID
│                    │                          header to the recipient
└──────┬─────────────┘
       │ Stalwart accepts, assigns its own queue id
       ▼
┌────────────────────┐
│ stalwartQueueId    │  217700302698266624     returned in the 250 line:
│                    │                         "250 2.0.0 Message queued
│                    │                          as 217700302698266624"
│                    │                         opaque integer string
└────────────────────┘
```

All three are persisted on the log row. The status endpoint joins on
`stalwartQueueId` because that's what the Stalwart admin API keys on.

## Status endpoint behaviour

`GET /v1/email/log/:id/status` returns one of:

| `state` | Meaning |
|---|---|
| `local_only` | Stalwart admin URL/token not configured. The row exists; delivery state is unknown. |
| `scheduled` | Stalwart has the message; next attempt pending. |
| `in_progress` | Stalwart is actively delivering. |
| `completed` | Stalwart confirms delivery to the recipient MX. |
| `temp_fail` | Soft failure (4xx, DNS, TLS, rate). Will retry until expiry. |
| `perm_fail` | Hard bounce (5xx, expired retries). Will not retry. |
| `completed_aged_out` | Stalwart returns 404 for the queue id _and_ the submit was >1h ago. Stalwart removes successfully-delivered messages from the queue, so this is _almost certainly_ a clean delivery — but not provable from the queue API alone. (The forensic record lives in `/api/telemetry/traces`; deferred to follow-up.) |
| `missing_queue_id` | We never parsed a queue id from the 250 line. Usually means the SMTP server returned a non-Stalwart-style response. |

`temp_fail` and `perm_fail` carry an `error_category` field from Stalwart
(`dns`, `tls`, `mta-sts`, `rate`, `unexpected-reply`, `connection`, `io`,
`dane`, `concurrency`).

## Feature flag — Stalwart proxy

```
STALWART_ADMIN_URL=https://mail.cloud-claude.com
STALWART_ADMIN_TOKEN=<bearer token issued by cloud-claude>
```

When **either** is unset, `/v1/email/log/:id/status` returns
`{ state: "local_only", reason: "stalwart_not_configured" }`. The submitted
record still works end-to-end. Setting both flips the endpoint live with no
code change.

The token is a multi-tenant blast radius — whoever holds it can read any
queued message body, reschedule, or cancel any send. Treat it like a database
admin password. The SPA never sees it.

## Privacy

- The submitted log stores the recipient email **in cleartext**. That's
  intentional — without it, the audit is useless for "who got the streak-save
  yesterday?" Access is gated by `requireUser` + `requireAdmin` (only emails
  in `SOCIAL_ADMIN_EMAILS`).
- The Stalwart admin API _also_ returns recipient addresses. Same gate.
- `subjectRendered` is stored in full. `bodyRendered` is **not** stored —
  intentionally — to keep the log small and to limit data-at-rest exposure.

## Operational runbook

### To provision Stalwart admin access (one-time)

1. Ask cloud-claude support to issue an admin Bearer token scoped to the
   `learnai@useyl.com` send identity on `mail.cloud-claude.com`.
2. Set on the LearnAI resource:
   ```
   STALWART_ADMIN_URL=https://mail.cloud-claude.com
   STALWART_ADMIN_TOKEN=<the bearer token>
   ```
3. Redeploy (or wait for next push; env-only changes don't need a redeploy
   on cloud-claude).
4. Verify in the admin tab: the "Delivery" column should populate within
   a few seconds for any pending row.

### To inspect the raw log

```sh
# from inside the container
tail -50 /data/email-log.jsonl | jq .
```

Each line is a complete JSON object. Lines are append-only and immutable.

### To answer "what did we send between dates X and Y?"

The local log is the first place to check (and the only one if the operator
ran the admin tab in this window). For the upstream source of truth —
including any direct-SMTP bypass — query Stalwart telemetry:

```
GET https://mail.cloud-claude.com/api/telemetry/traces
  ?type=delivery.attempt-start
  &from=<ISO>&to=<ISO>
  &values=return_path:learnai@useyl.com
```

Stalwart traces survive past the queue lifetime; that's where the historical
record lives.
